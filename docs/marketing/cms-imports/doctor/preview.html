The pitch

What this instance demonstrates

VBWD Doctor is a worked example of running the VBWD platform for private doctors, dentists, physiotherapists, therapists and small clinics. VBWD core is intentionally agnostic — all domain behaviour is delivered by plugins. This instance is nothing more than a chosen constellation of plugins on top of the same core codebase that every other instance runs.

Philosophy

Plug-and-play at the boundary

VBWD is deliberately open at the edges: any third-party system can be reached through a plugin adapter, outbound webhooks or the REST API. The list below is what VBWD Doctor typically connects to.

Ecosystem

Integrations for this vertical

SMS reminders — material no-show reduction in every study.

Appointment confirmations and pre-visit intake forms.

Telehealth link auto-generated per online appointment.

Webhooks

Push appointment events into any PMS or EHR.

Generic

Integration surfaces available everywhere

REST API

65+ endpoints under /api/v1/*. Auth via JWT, machine-to-machine via API keys (planned).

Outbound webhooks

Fire on every subscription and payment event. Configurable endpoints per event type, retry with backoff, delivery log.

Plugin adapters

Wrap any third party as a backend plugin under plugins/<name>/. Follows the stripe or mollie reference.

Zapier / n8n / Make

Consume webhooks and drive inbound actions through the REST API without writing code.

Architecture

How a payment plugin is structured

Each provider lives under plugins/<provider>/ with its own config schema, models, service layer, routes and tests. They all register against the same IPaymentAdapter interface — switch providers and nothing else in the codebase needs to move.

Need a rail we don't cover yet? Payment plugins are standalone repos; build your own using plugins/stripe/ as a reference or commission us →
