2 · Buy in the browser (live fe-user checkout)

Tapping /checkout mints a one-time draft link. Opening it loads the existing unregistered PublicCheckoutView, which fetches the draft and hydrates the fe-core cart — Total €64 (Pro €29 + Priority Support €15 + 5,000 tokens €20), server-recomputed from the catalogs. The bot creates no charge; the user signs in / registers and pays here.
What just happened (all real):
meinchat msg in bot convo → MeinchatInboundHook (IPostSendHook) // bot is a participant → bot-base UpdateDispatcher // provider-neutral → subscription BotCommandProvider // /tarifs /add-ons /tokens /checkout → bot_checkout_draft (generic line items) → GET /public/checkout-draft/<token> // recompute prices → fe-user PublicCheckoutView ?draft= // hydrate cart → pay

Why it matters

The same subscription storefront consumer drives this over meinchat with zero consumer change — it is written on the bridge’s provider-neutral DTOs, so it lights up over Telegram (and any future provider) identically. Identity is automatic on meinchat (the sender is the authenticated vbwd user), so billed reads like the live token balance work with no link step. The checkout draft is single-use, TTL’d, carries no identity or prices, and the bot never creates a charge — payment happens in the normal browser checkout.
