CMS pages render confirmation

2026-06-13 · Every previously-blank public CMS page now renders. Verified live on http://localhost:8080; screenshots captured with Playwright.

What was wrong: 16 of 40 published CMS pages rendered as a blank white screen. Their explicitly-assigned layouts (home-v1, ghrm-software-catalogue, booking-*, native-pricing-page, …) had zero widget placements and no content area, so nothing rendered.

Two-part fix:
  1. Graceful fallback (code): a page/post whose assigned layout is unseeded (0 cms_layout_widget rows) now falls back to the active default layout instead of a blank page — "no layout → default; not seeded → default." (--plugin cms --full green.)
  2. Seeded the real placements (data): each layout's widgets are now placed in their areas (hero/features/cta for home, GhrmCatalogueContent for the GHRM catalogue, the booking widgets, NativePricingPlans for native pricing). A page with its own seeded layout uses that layout (it overrides the default).

Conclusions

  • All 16 previously-blank pages now render — the widget-driven ones via their own seeded layouts (which override the default once seeded), and features via its page body.
  • The unseeded-layout → default fallback remains as the safety net: any future page whose layout has no placements degrades to the default chrome instead of a blank screen.
  • booking-form / booking-resource-detail show "Resource not found" when visited without a resource in the URL — this is the widget's correct context-required behaviour, not a placement gap.
  • Durability note: these placements live in the cms_layout_widget table (dev DB). Now that the cms_layouts exchanger round-trips widget_assignments, the placements should be baked into each plugin's layout import files (as done for shop) so a cold-start import reproduces them without manual seeding.
