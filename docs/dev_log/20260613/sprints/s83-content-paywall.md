# S83 — Content paywall (pay-per-view with tokens; CMS `<vbwdpaywall>` tag, plugin trio)

**Area:** NEW plugin trio — backend `paywall` (`vbwd-backend/plugins/paywall`, source in `plugins/paywall/paywall/` per the new convention) + fe-admin `paywall-admin` (`vbwd-fe-admin/plugins/paywall-admin`) + fe-user `paywall` (`vbwd-fe-user/plugins/paywall`); standalone repos `vbwd-plugin-paywall` / `vbwd-fe-admin-plugin-paywall` / `vbwd-fe-user-plugin-paywall` ([[feedback_plugins_always_in_own_repos]], [[feedback_work_in_sdk_dirs]]). **Depends on:** backend `cms` plugin (`PostService` public read; a NEW generic content-filter registry added here — `PluginMetadata.dependencies=["cms"]`); **core** `TokenService` (`current_app.container.token_service()` — token wallet is core, no dependency declaration); fe-admin `extensionRegistry` (`sectionItems.sales` sidebar injection, the booking/subscription-admin pattern); fe-user `cms` post render (a NEW minimal content-augmentation seam added here).
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** (narrowest change that satisfies the req) · drop deprecated code in touched paths — **BINDING per [`_engineering_requirements.md`](_engineering_requirements.md)**. Quality guard: `bin/pre-commit-check.sh --plugin paywall --full` green (+ full run because `cms` is touched — the new content-filter registry) + fe-admin & fe-user Vitest/ESLint green. **Not committed** without explicit instruction ([[feedback_no_commit_without_ask]]).

## Problem

There is no way to monetise individual pieces of CMS content. We want an editor to drop a single tag into any post or page —

```html
<vbwdpaywall id="paywall-slug" />
```

— and have **everything from that tag onward become locked**. A logged-in reader unlocks it **instantly by spending tokens** from their balance; a reader who is **not logged in** sees a friendly box: *"The content is hidden — log in and pay per view, or subscribe."* Admins manage paywalls from a new **Sales → Paywall** page with two tabs (**Paywalls**, **Categories**, where categories behave exactly like tarif-plan categories). A paywall has a **slug, public name, language, price in tokens, and a discounted price**.

The platform already provides every primitive: the token wallet is **core** (`TokenService.debit_tokens(user_id, amount, USAGE, reference_id, description)` — atomic, writes a `vbwd_token_transaction` ledger row, raises `ValueError` on insufficient balance), `cms_post.content_html` is **not sanitised** (a custom tag survives verbatim), tarif-plan **categories** + the **two-tab Plans page** + the **`sectionItems.sales` sidebar injection** are proven patterns to mirror, and IoC via **registries** is the house style (`data_exchange_registry`, `line_item_registry`, `post_type_registry`). The only genuinely new mechanics are: the **paywall catalogue** (paywall + category models, mirroring tarif plans), the **server-side gating** (a generic CMS content-filter the paywall populates — so gated bytes never reach an unpaid client and `cms` never imports `paywall`), the **instant token unlock** endpoint, and the **box + player swap** on the public page.

## Locked decisions

| # | Decision |
|---|----------|
| **D1 — `Paywall` is a catalogue entity mirroring `TarifPlan`; pricing is in whole tokens.** | `Paywall` model (`paywall` table): `slug` (unique, indexed — the tag's `id`), `public_name`, `language` (String(8), default `en`), `price_tokens` (Integer ≥ 0), `discounted_price_tokens` (Integer, nullable), `is_active` (default true), `sort_order`. **Effective price** = `discounted_price_tokens` when it is set **and** `0 ≤ discounted < price_tokens`, else `price_tokens` — a single `effective_price_tokens` helper used everywhere (DRY). Tokens are integers (the wallet `balance` is `Integer`) — no `vbwd_price`/currency object (that is fiat money; this is the token wallet). |
| **D2 — Categories are `TarifPlanCategory` cloned.** | `PaywallCategory` model (`paywall_category` table): `name`, `slug` (unique), `description`, `parent_id` (self-FK, hierarchical), `is_single`, `sort_order`, + an M2M `paywall_category_paywalls` to `Paywall` (the exact `tarif_plan_category` shape + service/repo/route surface). "Categories same like on tarif plans." |
| **D3 — The wallet is core; debit directly, instantly, idempotently.** | Unlock debits via the **core** `TokenService.debit_tokens(user_id, effective_price_tokens, TokenTransactionType.USAGE, reference_id=<paywall_access.id>, description="Unlocked paywall <slug>")`. **No invoice, no `token_payment` plugin** (that plugin pays *invoices*; this is a direct wallet spend). Insufficient balance → `ValueError` → `402`/`400` with a clear message. **No dependency declaration** (core infra). |
| **D4 — Unlock is permanent per (user, paywall) and double-charge-proof.** | `PaywallAccess` model (`paywall_access` table): `user_id` (core FK), `paywall_id` (FK → `paywall`), `unlocked_at`, `price_paid_tokens`, `token_transaction_id` (nullable). **Unique `(user_id, paywall_id)`.** "Pay per view" = pay once per paywall unit, then permanent access (re-reading never re-charges). The unlock flow checks existing access first **and** relies on the DB unique constraint as the race guard (catch `IntegrityError` → already unlocked → no second debit). No cross-plugin FK into `cms_post` — the binding is the tag's slug, not a post id ([[project_migration_graph_fragmentation]]). |
| **D5 — Gating is server-side via a NEW generic CMS content-filter registry; `cms` stays paywall-agnostic.** | Add `cms_public_content_filter_registry` to the `cms` plugin: filters are `(post_dict, current_user_or_none) -> post_dict`, applied in the public single-post route (`GET /api/v1/cms/posts/<slug>`) **only** (listings already return excerpts, never the body). `cms` knows "content filters", never "paywall" (no `from plugins.paywall`). The `paywall` plugin **registers its filter in `on_enable`** ([[project_plugin_di_provider_registration]]). This is the established IoC pattern (`data_exchange_registry` et al.). |
| **D6 — The paywall filter truncates at the FIRST tag and never ships gated bytes.** | The filter scans `content_html` for the first `<vbwdpaywall …/>` (lenient regex: self-closing or paired, `id="…"`/`id='…'`, any attribute order). Content **before** the tag is free; content **from the tag onward is the gated unit** (one boundary per post — multiple tags = the first wins; multi-zone is overengineering). If the resolved `Paywall` is missing/inactive → tag stripped, content shown free (fail-open on a dangling tag, logged). If the viewer **lacks access**: the body is cut at the tag, the tag replaced by a sentinel `<div class="vbwd-paywall-placeholder" data-paywall-slug="…"></div>`, and a top-level `paywall` object is attached to the post JSON: `{slug, public_name, price_tokens, discounted_price_tokens, effective_price_tokens, has_access:false, authenticated:<bool>}`. **The gated HTML is absent from the response** (a determined unpaid client cannot read it). If the viewer **has access**: the tag is stripped and the full body returned (`paywall.has_access:true`). |
| **D7 — Anonymous vs logged-in box copy, server-driven CTAs.** | fe-user renders the box from the `paywall` object where the sentinel sits. **Anonymous** (`authenticated:false`): *"The content is hidden — log in and pay per view, or subscribe."* + **Log in** (→ login, returns to the post) and **Subscribe** (→ `/plans`) CTAs. **Logged-in, no access:** shows `public_name` + **effective price in tokens** + current **balance**, an **Unlock for N tokens** button, and (if balance < price) a **Buy tokens** CTA + the Subscribe upsell. |
| **D8 — `<PaywallBox>` lives in the fe-user `paywall` plugin; integrate via a NEW minimal CMS render seam.** | Keep the box in the `paywall` plugin (fe-user `cms` must not own paywall UI). Add a small generic **post-body augmentation** seam to the fe-user `cms` plugin (a registry mirroring fe-admin's `extensionRegistry.dataExchangeTabs` / `invoiceDetailSections`): when `post.paywall && !post.paywall.has_access`, the registered `paywall` augmentation renders `<PaywallBox>` in place of the `.vbwd-paywall-placeholder` (the post still renders through `CmsLayoutRenderer` with its layout/style/widgets). The unlock button calls the unlock endpoint, then re-fetches the post (now full content). |
| **D9 — Instant unlock endpoint; price is server-authoritative.** | `POST /api/v1/paywall/<slug>/unlock` (auth required, any logged-in user — no special permission): resolve active `Paywall`; if `PaywallAccess` exists → `200 {already:true, balance}`; else debit `effective_price_tokens` (server-computed; the client never sends a price) → create `PaywallAccess` (unique-constraint guarded, D4) → `200 {unlocked:true, balance}`. Insufficient → `402`. Inactive/unknown slug → `404`. Read helpers: `GET /api/v1/paywall/<slug>` (public name/price/effective/`has_access` for the current session) for the box; admin CRUD under `/api/v1/admin/paywalls/…` + `/api/v1/admin/paywall-categories/…` (clone of the tarif-plan-category routes). |
| **D10 — "Subscribe" is an upsell CTA in v1; subscription auto-unlock is config-gated (default off).** | The spec's paywall fields contain **no** subscription link, so v1 gating is **token pay-per-view only**; the **Subscribe** CTA navigates to `/plans`. A `config.json` flag `subscription_unlocks_all` (default **false**) reserves the future behaviour: when true, any ACTIVE/TRIALING subscription unlocks all paywalls (resolved through an optional subscription read port, only if present — no hard `subscription` dependency). Implementing the bypass is **deferred** (§Deferred); the flag + CTA are the seam. |
| **D11 — Admin authoring: insert the tag, pick the paywall.** | The admin types/inserts `<vbwdpaywall id="slug" />` in the post/page **HTML tab** (raw; the WYSIWYG/Visual tab may strip unknown elements — documented, so authoring is via HTML). fe-admin adds a small **"Insert paywall"** helper in the HTML toolbar: a paywall-slug picker (lists active paywalls) that inserts the canonical tag at the cursor (nice-to-have; the literal tag always works). Permissions: new `paywall.manage` / `paywall.view` (admin), mirroring `booking.*`; `debug_mode` baseline in config ([[feedback_plugin_baseline_config_files]]). |

## Scope

### Slice 1 — backend `paywall` plugin (catalogue + access + unlock)
- Plugin class (`__init__.py`, `dependencies=["cms"]`, `on_enable` registers the content filter + DI providers per [[project_plugin_di_provider_registration]]), `config.json` + `admin-config.json` (`debug_mode`, `subscription_unlocks_all=false`), registration in `plugins/plugins.json` + `plugins/config.json`, migration (own branch, `plugins/paywall/migrations/versions/`, [[feedback_plugin_migrations_in_plugin]]) for `paywall`, `paywall_category`, `paywall_category_paywalls`, `paywall_access`.
- Models (D1/D2/D4) + repositories + `PaywallService` (`effective_price_tokens`, CRUD, `unlock(user_id, slug)` per D9 — atomic debit + access row, idempotent) + `PaywallCategoryService` (clone of `TarifPlanCategoryService`: create/update/delete/attach/detach, slug auto-gen, parent-cycle guard).
- `PaywallContentFilter` (D6): the `(post_dict, user) -> post_dict` callable registered into the new cms registry; pure + unit-testable (regex parse + truncate + attach `paywall` object).
- Routes (D9): public `GET /api/v1/paywall/<slug>`, `POST /api/v1/paywall/<slug>/unlock` (auth); admin `paywall.manage`-gated CRUD for paywalls + categories (clone tarif-plan-category route surface).
- `populate_db.py`: idempotent demo — 1 category + 1 paywall (`demo-paywall`, e.g. 5 tokens, discounted 3) and a demo post containing the tag, so the flow is visible.

### Slice 2 — backend `cms` plugin (the one generic seam)
- Add `cms_public_content_filter_registry` + apply it in `public_get_post` (D5). Generic, paywall-unaware, ordered, no-op when empty. Unit test: a registered filter is applied; none registered → response unchanged (Liskov/back-compat).

### Slice 3 — backend data-exchange (catalogue is portable, like everything else)
- `paywall_paywalls` + `paywall_categories` exchangers (cluster `sales`, perms `paywall.view`/`paywall.manage`), mirroring the subscription category + M2M-slug exchangers (categories carry `parent_slug`; paywalls carry `category_slugs`, resolved on import). `PaywallAccess` is **user data, never exported** as catalogue.

### Slice 4 — fe-admin `paywall-admin` plugin
- Sidebar (D11): `extensionRegistry.register('paywall-admin', { sectionItems: { sales: [{ label:'Paywall', to:'/admin/paywall', id:'paywall', requiredPermission:'paywall.view' }] } })` + translations (all locales).
- **`Paywall.vue`** two-tab page (mirror `subscription-admin/Plans.vue`): **Paywalls** tab (table: name · slug · language · price · discounted · active; search/sort/bulk) + **Categories** tab (lazy `PaywallCategoriesTab.vue`, clone of `CategoriesTab.vue`).
- **`PaywallForm.vue`** (mirror `PlanForm.vue`): slug (auto-gen, locked on edit) · public name · language · `price_tokens` · `discounted_price_tokens` + category assignment (Available/Assigned panels). **`PaywallCategoryForm.vue`** clones `CategoryForm.vue`.
- HTML-tab **"Insert paywall"** helper in the cms-admin post editor (D11) — contributed via the editor's existing plugin-tab/extension seam; picks an active paywall slug and inserts the tag.

### Slice 5 — fe-user `paywall` plugin
- Add the minimal **post-body augmentation** registry to fe-user `cms` (D8) and register the `paywall` augmentation.
- **`PaywallBox.vue`** (D7): anonymous vs logged-in states, effective price + balance, **Unlock** (→ `POST /unlock` → re-fetch post), **Buy tokens** / **Subscribe** CTAs; tokens-as-currency labelling; accessible + styled via fe-core tokens.
- Pinia store: `fetchPaywall(slug)`, `unlock(slug)`; balance read from the existing token-balance source.

### Deferred (explicitly out of scope)
Subscription **auto-unlock** (D10 flag is the seam, behaviour deferred) · multiple paywall zones per post / nested paywalls · time-limited or metered access (access is permanent once paid) · refunds/chargebacks of unlocks · fiat-priced paywalls (tokens only) · per-paywall analytics/conversion dashboards · gating inside the WYSIWYG/Visual editor (authoring via HTML tag) · gifting/sharing unlocks · gating non-CMS content (shop/booking) — paywall is CMS-content only this sprint.
