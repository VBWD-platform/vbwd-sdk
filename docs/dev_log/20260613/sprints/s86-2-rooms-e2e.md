# S86.2 — E2E for rooms: per-recipient fan-out in meinchat-plus ("encrypted unless invited")

**Parent:** [S86 umbrella](s86-bot-widget.md) (inherits AD1–AD10, esp. **AD5/AD6**). **Area:** `vbwd-backend/plugins/meinchat_plus` + `vbwd-fe-user/plugins/meinchat-plus`. **Depends on:** **S86.1** (rooms + membership + the protocol-selection seam + `meinchat_message.room_id`); the existing meinchat-plus Signal stack — device-key registry, signed/one-time prekeys + bundle endpoint, X3DH (`crypto/x3dh.ts`), double ratchet (`crypto/ratchet.ts`), envelope (`crypto/envelope.ts`), per-device `message_delivery`, `ProtocolAwareCodec`/`SignalEnvelopeValidator`, `BothPeersHaveDeviceKeys`, `E2eV1Capability`, the six meinchat extension ports.
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** — **BINDING per [`_engineering_requirements.md`](_engineering_requirements.md)**. Quality guard: `bin/pre-commit-check.sh --plugin meinchat_plus --full` (+ `--plugin meinchat --full` if the shared seam is touched) + `vbwd-fe-user` Vitest/ESLint green. **Not committed** without explicit instruction ([[feedback_no_commit_without_ask]]).

## Problem

S86.1 ships rooms on the **plain** (server-readable) protocol. Requirement 2 — **rooms are encrypted unless invited; only a member can read the content** — needs the meinchat-plus E2E layer to cover the N-party case. The server must still **hold no keys and never decrypt** (the plus invariant). The minimal way to get group E2E without inventing a new crypto primitive is **per-recipient fan-out over the pairwise sessions that already exist** (AD6): the sender encrypts each room message once per current member-device, the server stores per-device deliveries (already shipped), and a newly-invited member only ever receives fan-outs sent after they joined — which **is** "encrypted unless invited" (no plaintext, and no ciphertext, of pre-join history).

## Locked decisions

| # | Decision |
|---|----------|
| **D1 — Group E2E = per-recipient fan-out over existing pairwise Signal sessions; NO sender-keys / MLS / shared room key.** | A room `e2e_v1` message is the same opaque envelope as a 1:1 message, produced once per recipient member-device using the existing X3DH-established + double-ratcheted pairwise session. The server stores N per-device deliveries via the existing `message_delivery` path. Rationale (anti-overengineering): reuses 100% of the shipped crypto + delivery; bounded room sizes make O(members×devices) fan-out fine. Sender-keys/MLS + key-history-on-invite are explicitly **deferred**. |
| **D2 — `AllMembersHaveDeviceKeys` policy generalises `BothPeersHaveDeviceKeys` (Liskov).** | A new `IConversationPolicy`/room-policy impl vetoes starting/upgrading a room to `e2e_v1` if **any** member lacks an active device key, surfacing a `member_has_no_device_keys` contract (UI: "Ask @bob to enable secure chat"). Plain rooms stay allow-all (behaviour-preserving substitute). Consumes S86.1's generalised protocol seam (AD5). |
| **D3 — A room with a BOT member is never `e2e_v1`.** | Enforced already by AD5/S86.1 (bot present ⇒ plain). The plus policy additionally refuses to upgrade such a room. So the widget-bot rooms (S86.3) are always plain; E2E applies to **human-only** rooms. |
| **D4 — "Encrypted unless invited" = forward-only membership-scoped fan-out; no history re-encryption.** | On `invite`, the new member's device(s) join the fan-out recipient set for **future** messages only. Past ciphertext was never addressed to them and is not re-sealed (deferred). Server-side this needs no key handling — the server holds no keys; it just delivers to current-member devices. |
| **D5 — Server validates envelope shape/size only; capabilities advertise `e2e_v1` for rooms.** | `ProtocolAwareCodec` validates each fanned-out envelope exactly as for 1:1 (plain → passthrough). `E2eV1Capability.for_conversation` is extended to also answer `for_room` (or a sibling) so meinchat's capability union offers `e2e_v1` to all-human rooms. No new server crypto. |
| **D6 — fe-user (meinchat-plus) does the group encrypt/decrypt over the room member-device directory.** | The composer fetches the current room members' device bundles (reuse the device directory + prekey-bundle endpoint), establishes/loads a pairwise session per device (existing `session.ts`/`x3dh.ts`/`ratchet.ts`), and emits one envelope per device (reuse `crypto/messaging.ts` + `envelope.ts`). Decryption is unchanged (each member decrypts the envelope addressed to its own device). A composer-precheck generalises `useComposerPrecheck` to "all member devices have keys." |

## Scope
- **backend `meinchat_plus`:** `AllMembersHaveDeviceKeys` room policy (D2/D3) registered against meinchat's policy port; extend the capability advertiser for rooms (D5); confirm `ProtocolAwareCodec`/`SignalEnvelopeValidator` accept the room fan-out unchanged (they should — same envelope); per-device delivery for room messages rides the existing `message_delivery` (S86.1 made messages room-aware). No new model unless delivery needs a `room_id` denormalisation (only if a query requires it — justify before adding).
- **fe-user `meinchat-plus`:** room member-device fan-out on send (D6) — gather member device bundles, per-device session + envelope, one ciphertext per device; generalised composer-precheck; the secure-chat gate (`SecureChatGate.vue`) covers room entry (shows the "enable secure chat" prompt when a member lacks keys); decryption path reused as-is.
- **fe-user `meinchat` (S86.1 view):** the room view consumes whichever protocol the room pinned — `plain` renders as today; `e2e_v1` routes send/receive through the plus provider (the existing protocol-aware provider seam).

## TDD (tests first)
- **Backend unit:** `AllMembersHaveDeviceKeys.may_start` allows when every member has a key, vetoes (`member_has_no_device_keys`) when one lacks it, and allow-throughs for `plain`; refuses `e2e_v1` when a BOT is a member (D3); the codec validates a room fan-out envelope and rejects a malformed one.
- **Backend integration** (real PG): a 3-human-member `e2e_v1` room — a sent message produces one stored per-device delivery per recipient device, **zero plaintext on the server**; an invited 4th member receives fan-outs for **new** messages only and has no delivery rows for pre-join messages (D4); a room with a BOT member is forced `plain` and never offered `e2e_v1`.
- **fe-user (Vitest):** the composer fans out N envelopes for N member-devices (one per device, mock crypto); the precheck blocks send + shows the enable-secure-chat prompt when a member device lacks keys; a received room envelope decrypts via the existing path; a `plain` room bypasses encryption.

## Definition of done
Human-only rooms run **`e2e_v1`** via per-recipient fan-out over the existing pairwise Signal sessions; **the server holds no keys and stores only opaque envelopes** (one per recipient device); **only members can read** and a member can read only messages from **after their invite** ("encrypted unless invited"); a room with any BOT member is forced `plain` (S86.3 widget rooms). No new crypto primitive; history-on-invite and sender-keys/MLS deferred. `bin/pre-commit-check.sh --plugin meinchat_plus --full` + fe-user Vitest/ESLint green. Not committed without explicit instruction.
