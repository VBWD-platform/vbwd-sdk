# S86.1 — Rooms foundation: core `GUEST` role + base meinchat rooms (multi-party, plain)

**Parent:** [S86 umbrella](s86-bot-widget.md) (inherits AD1–AD10). **Area:** `vbwd-backend` core (`vbwd/models/enums.py` + auth/RBAC) + `vbwd-backend/plugins/meinchat` + `vbwd-fe-user/plugins/meinchat` + `vbwd-fe-admin/plugins/meinchat-admin`. **Depends on:** the existing meinchat conversation/message/nickname stack, the Redis event bus + SSE stream, the conversation UI. **Declares:** nothing new cross-plugin (rooms are base meinchat).
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** — **BINDING per [`_engineering_requirements.md`](_engineering_requirements.md)**. Quality guard: `bin/pre-commit-check.sh --full` (core IS touched, D-core) **and** `--plugin meinchat --full` + fe-user/fe-admin Vitest/ESLint/vue-tsc green. **Not committed** without explicit instruction ([[feedback_no_commit_without_ask]]).

## Problem

meinchat conversations are strictly **2-party** — the `Conversation` table is pair-keyed (`participant_low_id < participant_high_id`, unique pair, no-self check). There is no way to have a conversation with three or more participants. We need **rooms**: an N-party conversation with **membership, roles, and invites** (AD1, AD3, AD4), running on the existing message/stream machinery, on the **plain** protocol (E2E is layered on in S86.2). We also add the core **`GUEST`** identity (AD7) here, because rooms are the first surface a guest participates in (the widget, S86.3, needs it).

| **D6 — Retention covers rooms.** | The existing retention service/scheduler (TESTING-guarded) prunes room messages by the same policy as conversation messages (rooms reuse `meinchat_message`, so this is mostly free); a deleted room CASCADEs members + messages. |

## Scope
- **Slice 0 (core):** `UserRole.GUEST` + auth guard + RBAC public-scope mapping + (if needed) the enum-add core migration. Tests: GUEST can't password-login; GUEST token rejected on a user-scope route, accepted on a public-scope route.
- **Slice 1 (backend meinchat):** D1/D2 models + migration; `RoomService` + `RoomRepository` + `RoomMemberRepository` (D3); `MessageService` room path (D2); protocol selection seam (D4); routes (D5); retention (D6); DI registration in `on_enable`.
- **Slice 2 (fe-user meinchat):** a `RoomConversationView` (reuse a shared `ConversationPane` factored out of `ConversationView` — DRY) + create-room + invite-member UI (member picker by nickname; "any member can invite" enabled for members, remove/rename gated to admin); room rows in the inbox list; `useMessagingStream` subscribes `room:<id>`.
- **Slice 3 (fe-admin meinchat-admin):** rooms appear in the existing moderation surface only as needed (no content inspection — privacy stance preserved); minimal (e.g. nothing, or a room count). Keep scope tight.

## TDD (tests first)

- **Backend unit** (`MagicMock`): `create_room` sets creator `admin` + others `member` and pins `plain` when a BOT is a member; `invite` succeeds for **any** member and 403s a non-member, idempotent on a repeat; `remove_member`/`rename` admin-gated; `leave` drops self; `get_for_member` 403s a non-member; message room-path stores a message with `room_id` and rejects both-null/both-set (CHECK contract).

- **fe-user (Vitest):** create-room flow posts member nicknames; the room view binds the shared pane + `room:<id>` stream; invite control is visible to a member and the remove/rename controls only to the admin; an inbox room row renders.
