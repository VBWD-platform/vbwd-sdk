# S86.3 — Bot-widget as a room: a CMS widget that creates a room on "Start Conversation"

**Parent:** [S86 umbrella](s86-bot-widget.md) (inherits AD1–AD10, esp. **AD7/AD8/AD9/AD10**). **Area:** `vbwd-backend/plugins/meinchat` (widget-start → room create; guest provisioning) + `vbwd-backend/plugins/bot_meinchat` (bots as room members) + `vbwd-fe-user/plugins/meinchat` (the CMS Vue widget) + `vbwd-fe-admin/plugins/meinchat-admin` (widget editor) + `vbwd-fe-admin/plugins/cms-admin` (promote the widget-editor registry to a shared seam). **Depends on:** **S86.1** (rooms + membership + `RoomService` + core `GUEST`); the CMS `vue-component` widget type + fe-user `vueComponentRegistry` (`registerCmsVueComponent`, cross-plugin); the fe-admin `widgetEditorRegistry`; the bot bridge (`bot_base` SPI + `bot_meinchat` provider, S45/S70) + portable bot style (`useBotConversationStyle`, S70.4); a provisioned BOT peer (the `assistant`). **S86.2 is NOT required** — widget-bot rooms are `plain` (AD5/AD9). **Declares:** meinchat → `cms` in `PluginMetadata.dependencies` (AD10).
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** — **BINDING per [`_engineering_requirements.md`](_engineering_requirements.md)**. Quality guard: `bin/pre-commit-check.sh --plugin meinchat --full` (+ `--plugin bot_meinchat --full` if touched) + fe-user/fe-admin Vitest/ESLint/vue-tsc green. **Not committed** without explicit instruction ([[feedback_no_commit_without_ask]]).

## Problem

With rooms in place (S86.1), the bot-widget becomes a **room auto-created for the visitor** (AD9). A CMS editor drops a `MeinchatChatWidget` into a layout, types a member list (`user1, user2` — nicknames, **any may be a bot**, e.g. a checkout/subscription bot + an LLM bot), and sets the visibility. When a page visitor clicks **"Start Conversation"**, the backend creates a room where the **visitor is the creator/admin** and the configured nicknames are **invited members**; because at least one member is usually a bot, the room is `plain` (AD5) and the bots reply through `bot_meinchat`. A logged-in visitor without a nickname is asked for one; a public visitor enters a name and joins **as a core GUEST** (AD7).

## Locked decisions

| # | Decision |
|---|----------|
| **D1 — `MeinchatChatWidget` is a CMS `vue-component` widget owned by the meinchat FE plugins (no cms code).** | fe-user `meinchat` registers `registerCmsVueComponent('MeinchatChatWidget', …)` in `install()` (the dynamic-import-of-the-cms-registry pattern booking/shop use). Stored widget: `widget_type='vue-component'`, `content_json.component='MeinchatChatWidget'`, `config={…}` (D3). |
| **D2 — Widget config is SERVER-TRUSTED via a narrow `ICmsWidgetReader` port; cms is a SOFT dependency.** | `ICmsWidgetReader.get_active_widget_config(slug) -> dict \| None` resolved from the registry/container; a `NullCmsWidgetReader` is the default (cms absent/disabled → `404 widget_not_found`, Liskov), and a cms-backed adapter (guarded import of `plugins.cms…CmsWidgetRepository.find_by_slug`) is wired only when cms is present. **meinchat does NOT hard-depend on cms** — messaging works fully without the CMS; only the widget-start endpoint needs cms. The member list + visibility come from the **stored** widget (`config`), never the request body — a public client cannot invite arbitrary users. (Refines AD10: soft, not hard, dependency.) |
| **D3 — Widget config shape (static; member list is text ids per [[reference_admin_config_select_static_only]]).** | `{ component_name:'MeinchatChatWidget', member_nicknames:string[] (editor types "user1, user2"), visibility:'public'\|'logged_in', room_name?, start_button_label?='Start Conversation', welcome_message?, composer_placeholder?, display?:'inline'\|'dock', open_by_default?:boolean }`. `member_nicknames` are **text ids** (no live directory dropdown). No secrets (public JSON). |
| **D4 — Visibility gate (AD7); a public widget's configured members must all be reachable-as-bots-or-public.** | `logged_in`: unauthenticated → "log in to chat" CTA. `public`: any visitor may start, but a public widget whose configured members include a **non-BOT human** is rejected at start (`409 public_human_member_not_allowed`) — publicly inviting a real person is deferred (§Deferred). So a public widget room = visitor (GUEST) + bots. |
| **D5 — "Start Conversation" creates a room with the visitor as admin + configured nicknames invited (AD9).** | `POST /api/v1/messaging/widget/start` (no `@require_auth`): body `{ widget_slug, display_name? }`. Resolve config (D2) → branch on visibility. **`logged_in`**: require a token (`401` if absent); caller without a nickname → `409 {code:'nickname_required'}` (FE → `PUT /nickname/me` → retry); call `RoomService.create_room(creator_id=caller, member_user_ids=resolved(member_nicknames), widget_slug=…)`. **`public`**: require `display_name` (`400` if absent); rate-limit by IP (`widget_guest_start`); provision a `GUEST` (AD7) with a uniquified nickname; mint a short JWT (reuse `TokenService`); create the room with the GUEST as creator/admin + the bots invited. Return `{ access_token?, room_id, self_nickname, members }`. Resolving an unknown member nickname → `404`. |
| **D6 — Bots participate as room members; `bot_meinchat` reacts to room messages addressed to a bot member.** | The bot inbound path (today fired on 1:1 message send) is extended to fire for a **room** message when a BOT is a member; the bot's `BotReply` posts back into the **room** (plain, S70 rich `meta` intact). The `MeinchatProvider` stays provider-neutral; only the in-process transport learns the room parent. |
| **D7 — Multi-bot routing in one room (CONFIRMED 2026-06-13).** | With two bots in a room (checkout bot + LLM bot), free text routes to one. **Rule (owner-confirmed):** reuse `bot_base`'s `BotSession` (active-plugin-owns-free-text) **scoped per room** — a `/command` routes to the owning bot's namespace; **free text falls back to the LLM bot** (the conversational default) whenever no other bot holds the active session. A `/checkout`-style command hands the room to the checkout bot until it releases (then free text reverts to the LLM bot). Richer arbitration (mentions, per-bot turn-taking) is **deferred**. |
| **D8 — fe-user widget = a thin shell over the room UI (D-reuse).** | `MeinchatChatWidget.vue` reads `props.config`, renders the permission gate + nickname/name prompt + a **"Start Conversation"** button, then on start binds the shared `ConversationPane` (S86.1) to `room_id`, subscribes the `room:<id>` stream, applies `applyBotConversationStyle` (S70.4), and shows `welcome_message` as the opening bubble. Honours `display`/`open_by_default`. Public stores `access_token` in memory (session-scoped, not localStorage). |
| **D9 — fe-admin editor via the EXISTING `widgetEditorRegistry`, promoted to a shared seam (OCP).** | `cms-admin` re-exports `registerWidgetEditor`/`getWidgetEditor`/`VueWidgetEditorDescriptor` from a stable path (re-export only — characterisation test asserts built-ins still register). `meinchat-admin` registers a `MeinchatChatWidgetEditorTab` (fields per D3; `member_nicknames` a comma-list input; `visibility` static select; `buildPreview` renders a static chat-window mock with a Start button). No meinchat strings in cms-admin; `CmsWidgetEditor.vue` stays widget-agnostic. |
| **D10 — Reuse permissions; widget rooms are stored (AD8).** | Editor reached via existing `cms.manage`; public start IP-rate-limited; GUEST bounded by core RBAC (S86.1). No new permission. No ephemeral relay — the room is a normal stored room, membership-gated. |
| **D11 — The widget-bot is metered: 1 token = 1 word (REVISED 2026-06-14, owner-confirmed).** | On **chat activation** (the public `widget/start`) the guest is granted `guest_initial_tokens` (admin-set, the only knob that matters now) via the core `TokenService` — **idempotent** (a returning guest reused per D12 is NOT re-granted). **Billing: 1 token = 1 word** (`guest_token_cost_per_word`, default `1`). Both the guest's **question** words and the bot's **answer** words are charged. Because an answer's word count is only known once the bot replies, charging is a **post-send hook**: after any message in a widget room (guest OR bot), debit `word_count × cost_per_word` from the room's guest. **Gate = balance positive:** a guest may send only while balance `> 0`; a send when balance `≤ 0` is blocked with `402 {code:'insufficient_tokens'}` (no message, bot not triggered). The in-flight round completes (spend-until-empty); the NEXT send is gated. The widget shows a **"Buy tokens to continue dialogue"** block; its link target is the future public page **`/tokens`** (not built yet — placeholder; the block is the stop signal for now). No new ledger (core token balance). Plugin-config gated (`guest_economy_enabled`, default on). **Replaces** the earlier flat `guest_question_token_cost`/`guest_answer_token_cost` round model. |
| **D12 — First-level overuse protection only; advanced fingerprinting deferred (NEW, owner-requested 2026-06-13).** | **Implement only level 1:** persist the guest identity client-side in BOTH a **persistent cookie** AND **long-lived `localStorage`** (the guest token + id), so a refresh / return **reuses the same guest + balance** — `widget/start` reuses a valid presented guest session instead of provisioning a fresh one and re-granting tokens (this is what makes D11's grant idempotent against simple reloads). **File-log only (NOT DB, NOT enforcement):** on `widget/start`, append the server-visible candidate browser/device-independent identification signals (IP, User-Agent, Accept-Language, …) to a log file so overuse patterns can be analysed later. **Deferred (discuss later):** a real device/browser-independent fingerprint (keyboard/connection/traceroute/footprint hash) + actual enforcement against incognito / cleared-cookie abuse. Level 1 is acknowledged best-effort (cookies/localStorage are clearable). |

## Scope
- **backend `meinchat`:** `ICmsWidgetReader` port + cms-backed impl + null impl (D2); declare `cms` dep; `GuestSessionService` (provision `GUEST` user + uniquified nickname + scoped JWT, D5/AD7) mirroring `BotSenderProvisioner`; `POST /messaging/widget/start` (D5, thin route → service) + `widget_guest_start` rate-limit; `populate_db.py` seeds one demo **public** `MeinchatChatWidget` with members `[assistant]` (or `[assistant, <llm-bot>]`) through the service/widget layer ([[feedback_no_direct_db_for_test_data]]).
- **backend `bot_meinchat` / `meinchat`:** extend the bot inbound transport to fire for room messages with a BOT member and post replies into the room (D6); per-room `BotSession` routing (D7).
- **backend token economy (D11):** grant `guest_initial_tokens` on activation via core `TokenService` (idempotent per guest); debit on guest question + before bot answer in the widget room flow; insufficient balance → blocked with `code:'insufficient_tokens'`; config keys `guest_economy_enabled`/`guest_initial_tokens`/`guest_question_token_cost`/`guest_answer_token_cost` in `config.json`+`DEFAULT_CONFIG`+`admin-config.json`. The "Buy tokens" action reuses the existing token-bundle purchase flow.
- **backend overuse protection level-1 (D12):** `widget/start` reuses a valid presented guest session (cookie/localStorage token) instead of re-provisioning + re-granting; a small fingerprint-candidate **file logger** (IP/UA/Accept-Language) — log only, no DB, no enforcement.
- **fe-user `meinchat`:** `MeinchatChatWidget.vue` + register the CMS component (D1/D8); gates + Start button + nickname/name prompt; bind the room pane + stream + bot style; **persist the guest token in a cookie + long-lived localStorage and reuse it on return (D12); render the "Buy tokens to continue dialogue" block on `insufficient_tokens` (D11)** (+ optional remaining-balance display). Translations (8 locales).
- **fe-admin:** cms-admin shared-seam export (D9) + meinchat-admin `MeinchatChatWidgetEditorTab` + descriptor registration. Translations (8 locales).

### Deferred
Publicly inviting a **human** member (D4) · multi-bot arbitration beyond the D7 default · attachments in the widget · guest "claim my chat after signup" account-merge · CAPTCHA on guest start (IP rate-limit first) · per-widget `required_permission` finer than the two-mode `visibility` · **advanced device/browser-independent fingerprint + overuse enforcement (D12 — keyboard/connection/traceroute/footprint hash, incognito/cleared-cookie defence)** · per-widget token-economy override.

## TDD (tests first)
- **Backend unit** (`MagicMock`): widget-start rejects unknown/inactive widget (`404`), `logged_in`+anonymous (`401`), `logged_in`+no-nickname (`409 nickname_required`), `public` with a non-BOT human member (`409 public_human_member_not_allowed`, D4), an unknown member nickname (`404`); a valid `public` start provisions a GUEST, mints a token, and creates a room with the GUEST as `admin` + the bots as members; the cms-disabled null reader → `404` (Liskov); `widget_guest_start` rate-limit consulted; D7 routing — a `/command` routes to the owning bot, free text defaults to the LLM bot.
- **Backend integration** (real PG): a public Start → a room exists with the GUEST (admin) + `assistant` (member); the GUEST sends a message and the **assistant replies into the room** (plain, S70 meta); a second bot member also reacts per D7; the guest JWT authorises only its own room and is 403 elsewhere; demo widget seed is idempotent.
- **Token economy (D11):** activation grants `guest_initial_tokens` (idempotent — a returning guest reusing its session is NOT re-granted); a question debits `guest_question_token_cost` and a bot answer debits `guest_answer_token_cost`; when the balance can't cover the next step the send/answer is blocked with `insufficient_tokens`; `guest_economy_enabled=false` disables all debiting.
- **Overuse protection level-1 (D12):** presenting a valid guest session token reuses the same guest (no new grant); the fingerprint candidate signals (IP/UA/Accept-Language) are written to the log file and NOT persisted to the DB.
- **fe-admin (Vitest):** descriptor registers under `'MeinchatChatWidget'`; the editor binds `v-model:config` incl. the comma-list `member_nicknames`; `buildPreview` emits the static mock with a Start button; the cms-admin re-export still registers every built-in editor (characterisation).
- **fe-user (Vitest):** `logged_in`+unauthenticated → CTA; `logged_in`+no-nickname → nickname step → `PUT /nickname/me` → Start; `public` → name step → Start → `widget/start` → room pane bound to `room_id`; `welcome_message` opens the thread; bot style applied; `dock`+`open_by_default:false` collapsed.
