# S82 — Video hosting (CMS "video" post type: self-hosted upload + embed)

**Area:** `vbwd-backend/plugins/cms` (existing plugin → `src/` convention) + `vbwd-fe-admin/plugins/cms-admin` + `vbwd-fe-user/plugins/cms`. **No new plugin** — this is a CMS extension. **Depends on:** the CMS post-type registry (`src/services/post_type_registry.py`), the unified `cms_post` table (`type` discriminator + `type_data` JSON, S47.0), `IFileStorage` (`ManagerBackedFileStorage`, uploads namespace), the `cms_images` asset+exchanger pattern (the template this mirrors), and the fe-user layout renderer (`CmsLayoutRenderer.vue`). Declared `PluginMetadata` unchanged (still the `cms` plugin).
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** (narrowest change that satisfies the req) · drop deprecated code in touched paths — **BINDING per [`_engineering_requirements.md`](_engineering_requirements.md)**. Quality guard: `bin/pre-commit-check.sh --plugin cms --full` green (+ full run if any core file is touched — it should **not** be) + fe-admin & fe-user Vitest/ESLint green. **Not committed** without explicit instruction ([[feedback_no_commit_without_ask]]).

## Problem

The CMS can host pages and posts, but **there is no way to publish a video as first-class content**. An editor who wants a "video" page must hand-write an `<video>`/`<iframe>` tag in the HTML tab, with no managed upload, no poster, no transcript, no reusable media, and no import/export of the binary. We want a proper **"video" post type**: the editor either **uploads a video file that the platform hosts and serves** (with seeking) or pastes an **external embed URL** (YouTube/Vimeo/direct file), adds a poster image + optional transcript/captions, and publishes — and the public site renders a real HTML5 player (or a sanitised provider embed) inside the normal CMS layout/style system. The video binary must round-trip through the unified Import/Export like images do.

The architecture already supports almost all of this with **zero core changes and zero `cms_post` migration**: post types are a registry (`register_post_type`), per-type fields are declared on `PostType.type_data.fields[]` and rendered generically by the admin editor, type-specific metadata lives in the existing `type_data` JSON column, and `/uploads/<path>` is served by Flask `send_from_directory` (conditional → `Accept-Ranges: bytes`; nginx in prod) so **range/seeking works out of the box**. The only genuinely new pieces are: a **video asset** (so large binaries are managed/served/exchangeable, not stuffed into JSON), an **upload endpoint**, the **player rendering**, and an **exchanger** — each a direct mirror of the proven `cms_images` machinery.

## Locked decisions

| # | Decision |
|---|----------|
| **D1 — `video` is a registered, routable post type; no `cms_post` migration.** | On `cms` enable, `register_post_type(PostType(key="video", label="Video", routable=True, hierarchical=False, type_data={"fields":[…]}))`. Video metadata lives in the existing `type_data` JSON column of `cms_post`. The unified post table, `PostService`, and all existing post routes (`/api/v1/admin/cms/posts?type=video`, public `/api/v1/cms/posts?type=video`, `/api/v1/cms/posts/<slug>`) work unchanged. The post-type registry's existing fail-fast on unknown types still applies. |
| **D2 — Self-hosted binary is a managed asset (`CmsVideo`), mirroring `cms_images` — NOT raw bytes in `type_data`.** | New `CmsVideo` model (`cms_video` table, own plugin migration): `slug` (natural key, unique-indexed), `title`, `file_path`, `url_path`, `mime_type`, `file_size_bytes`, `duration_seconds` (nullable), `poster_image_url` (nullable), `created_at`/`updated_at` (BaseModel). A video **post** references the asset by **`type_data.video_slug`** (slug, never a raw path) so it round-trips like every other slug link. Rationale (anti-overengineering check): videos are large → base64-in-JSON export is unacceptable and a managed asset gives reuse + a media list + clean ZIP export. This is the **exact** `CmsImage` shape minus image-only fields (width/height) plus `duration_seconds`/`poster_image_url` — not new machinery, the established one. |
| **D3 — Two source kinds: `upload` (hosted) and `embed` (external).** | `type_data.source_kind ∈ {"upload","embed"}`. `upload` → `video_slug` (→ `CmsVideo`, served from `/uploads/...`). `embed` → `embed_url`. Both also carry optional `poster_image_url`, `transcript` (plain text), and `captions` (list of `{lang, url}` WebVTT). This satisfies "hosting" (upload) without reinventing a player for the embed case. |
| **D4 — No transcoding, no chunked/resumable upload (deferred).** | The plugin stores the uploaded file **as-is** via `IFileStorage` under a `videos/` path; one multipart `POST` (no resumable/tus, no worker). No ffmpeg: no transcode, no auto-thumbnail/duration probe — `duration_seconds` is optional (editor may type it) and the poster is an explicitly-picked image. Size bounded by config + nginx `client_max_body_size` (documented). Chunked upload, transcoding, and auto-poster are explicit fast-follow (§Deferred). |
| **D5 — Range/seeking is already supported — confirm, don't rebuild.** | Dev: `send_from_directory` is `conditional=True` → emits `Accept-Ranges: bytes` and answers `Range` with `206 Partial Content`. Prod: nginx serves `/uploads` directly with native range support. No new serving code — a test asserts a `Range: bytes=…` request returns `206` with the correct slice. |
| **D6 — Rendering is layout-driven + type-aware, mirroring the existing `postHeaderHtml` seam.** | fe-user: a small `CmsVideoBlock.vue` renders when `post.type === 'video'` — `upload` → `<video controls preload="metadata" :poster>` with `<source>` = `url_path` and `<track kind="captions">` per `captions[]`; `embed` → a **sanitised** provider embed (D9). It is injected into the layout's content area so video posts still use the CMS layout/widgets/style system. Transcript renders below the player (accessible, SEO-visible). Mirrors the one-conditional approach already in `CmsPage.vue` — no new rendering registry. |
| **D7 — fe-admin reuses the generic type-field renderer; extend it with `image`/`video`/`select`/`textarea` field types.** | The admin `PostEditor.vue` already renders `PostType.type_data.fields[]` generically (text/number) into `type_data`. Extend that one renderer to also handle `type:"select"` (source_kind), `type:"textarea"` (transcript), `type:"image"` (reuse `CmsImagePicker` → poster), and `type:"video"` (new `CmsVideoPicker` → upload/select an asset, emits `video_slug`). The video type's field list is then **declared in the registry (D1)** — no bespoke editor view. A thin `VideoList.vue` wrapper (mirror of `PostList.vue`) scopes the list to `type=video` with a "+ New Video" button + route + sidebar item. DRY: the new field types benefit any future post type. |
| **D8 — Data-exchange: new `cms_videos` exchanger mirroring `cms_images`.** | `cms_videos` entity (cluster `content`, perms `cms.pages.view`/`cms.pages.manage`): JSON carries the row + base64 `data` for small files, ZIP carries the binary as an `assets/` file (exactly `CmsImagesExchanger`'s dual mode), upsert by `slug`. `cms_posts` already serialises `type_data`, so `video_slug`/`embed_url`/poster/transcript travel with the post. Import order: **videos before posts** (same slug-link discipline as categories-before-resources in S61). |
| **D9 — Embeds & uploads are a security boundary.** | (a) Upload: validate by **content sniff (magic bytes)** not just the client `Content-Type`, against `allowed_video_mime_types`; enforce `max_video_size_bytes`; filename sanitised; stored inside the uploads namespace only (realpath-within-namespace, already enforced by `ManagerBackedFileStorage`). (b) Embed: `embed_url` is validated against a **provider allow-list** (YouTube, Vimeo) normalised to their canonical `/embed/` form, or a direct same-origin video file URL — arbitrary URLs/iframes are rejected (anti-XSS/clickjacking); the player iframe uses a restrictive `sandbox`/`allow`. No raw HTML from `type_data` is ever rendered unescaped. |
| **D10 — Permission & config reuse.** | All endpoints gated on the existing `cms.manage` (admin) / public read needs none — **no new permission**. New `config.json`/`admin-config.json` keys: `max_video_size_bytes` (default `209715200` = 200 MB), `allowed_video_mime_types` (default `"video/mp4,video/webm,video/ogg"`); videos served under the existing `uploads_base_url`. `debug_mode` baseline already present ([[feedback_plugin_baseline_config_files]]). Do **not** overload the image `allowed_mime_types` — videos get their own keys + endpoint. |

## Scope

### Slice 1 — backend `cms`: video post type + asset + endpoints
- **Register** the `video` post type (D1) with its `type_data.fields[]` schema (source_kind `select`, video `video`, embed_url `text`, poster_image `image`, transcript `textarea`, duration_seconds `number`).
- **`CmsVideo` model** (`src/models/cms_video.py`, D2) + repository (`src/repositories/`) + plugin migration in `plugins/cms/migrations/versions/` chained off **`20260606_1100_cms_layout_default`** (current head) — registered in `alembic.ini version_locations` ([[feedback_plugin_migrations_in_plugin]]).
- **`CmsVideoService`** (DI-wired in `on_enable` per [[project_plugin_di_provider_registration]]): `upload_video(bytes, filename, mime_type, *, title=None, duration_seconds=None, poster_image_url=None)` (sniff+size validation D9), `list_videos`, `get_video`, `delete_video`.
- **Routes** (D10-gated): `POST /api/v1/admin/cms/videos/upload` (multipart `file` + optional `title`/`duration_seconds`/`poster`), `GET /api/v1/admin/cms/videos`, `GET/DELETE /api/v1/admin/cms/videos/<id>`. Serving reuses the existing `/uploads/<path>` route (D5).
- **Config** (D10): add the two keys to `config.json` + `DEFAULT_CONFIG` + `admin-config.json` (Storage tab).
- `populate_db.py`: optionally seed one demo `embed` video post (no binary) so the type is visible idempotently; **no** binary fixture committed.

### Slice 2 — backend `cms`: data-exchange
- **`CmsVideosExchanger`** (`src/services/data_exchange/cms_exchangers.py`, D8) mirroring `CmsImagesExchanger` (JSON base64 + ZIP asset, upsert by slug); register `cms_videos` in `build_cms_exchangers`.
- Confirm `cms_posts` round-trips `type_data` (video_slug/embed_url/poster/transcript/captions) — add a regression test if not already covered.

### Slice 3 — fe-admin `cms-admin`
- **Extend the generic type-field renderer** in `PostEditor.vue` (D7) to support `select`/`textarea`/`image`/`video` field types (text/number already exist). `image` reuses `CmsImagePicker`; `video` uses a new **`CmsVideoPicker.vue`** (upload to `/admin/cms/videos/upload` or pick an existing asset → emits `video_slug`); store gets `uploadVideo`/`fetchVideos`.
- **`VideoList.vue`** wrapper (mirror `PostList.vue`) → `<CmsContentList type="video" />`, route (`cms-video-new`/list), sidebar item under the CMS section. Editor tab behaviour: video type → HTML tab default (like pages), since the body is the player block.
- Translations for all locales.

### Slice 4 — fe-user `cms`
- **`CmsVideoBlock.vue`** (D6/D9): HTML5 player for `upload`, sanitised provider embed for `embed`, poster, `<track>` captions, transcript; injected into the layout content area in `CmsPage.vue` when `post.type === 'video'`.
- Public route for video posts (`/video/:slug` or via the existing type-routing the way `post` is routed — match whatever `post` uses; confirm `resolve_published_path('video', slug)` resolves). No auth.

### Deferred (explicitly out of scope)
Server-side **transcoding** / multi-rendition / adaptive streaming (HLS/DASH) · **auto** thumbnail/poster + duration probe (ffmpeg) · **chunked/resumable** upload (tus) for very large files · signed/private/expiring video URLs + DRM · view/playback analytics · playlists/series · live streaming · S3/CDN storage backend (uses the existing local `IFileStorage`; the abstraction already allows a later swap) · subtitle authoring UI (captions are entered as URLs to pre-made WebVTT).

## TDD (tests first)
- **Backend unit** (`MagicMock` repos): `CmsVideoService.upload_video` rejects a disallowed mime (sniffed, even if the header lies) and an over-size file; accepts mp4/webm; builds correct `url_path`. `video` post create stores `type_data` for both `upload` (with `video_slug`) and `embed` (with allow-listed `embed_url`); a non-allow-listed `embed_url` is rejected (D9). `register_post_type` exposes `video` via the post-types listing.
- **Backend integration** (real PG): migration up/down/up for `cms_video`; upload → asset row + file on disk → `GET /uploads/...` returns `200` and a `Range: bytes=0-1023` returns `206` with the right slice (D5); `cms_videos` exchanger JSON + ZIP round-trip (export → import into a clean session recreates the row + binary, upsert by slug, no duplicate); a `video` post export→import carries `type_data.video_slug` and re-links (videos-before-posts).
- **fe-admin (Vitest):** the field renderer renders select/textarea/image/video controls from a `type_data.fields[]` schema and writes into `type_data`; `CmsVideoPicker` posts the upload and emits the returned `video_slug`; `VideoList` filters the post list by `type=video` and the create button carries `?type=video`.
- **fe-user (Vitest):** `CmsVideoBlock` renders an `<video><source>` + `<track>` for `upload`, a sanitised iframe for an allow-listed `embed`, **nothing dangerous** for a hostile `embed_url`; the block is injected into the layout content area for `type==='video'` and the post still renders layout/widgets/style.
