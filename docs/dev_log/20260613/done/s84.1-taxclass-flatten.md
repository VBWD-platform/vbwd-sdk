**Area:** **core** `vbwd-backend` (`Tax` model, `vbwd/routes/admin/tax.py`, `taxes` data-exchange exchanger, migration) + `vbwd-fe-admin` (Taxes tab) · **Depends on:** S72.1–72.4 (Taxes & Countries), the S72.3 entity→tax M2M decision. · **Origin:** a design question raised during [S84 (Currency)](s84-currency-admin-tab.md).
**Engineering requirements:** TDD-first · SOLID · DI · DRY · clean code · **NO OVERENGINEERING** — **BINDING** per [`../../20260525/sprints/_engineering-requirements.md`](../../20260525/sprints/_engineering-requirements.md). **Status:** 🟢 **DONE on disk, gate-green — 2026-06-13.** Full write-up: [report 10](../reports/10-taxclass-flatten-and-tax-delete-409.md).

## Problem
`TaxClass` (`vbwd_tax_class`: name, code, description, default_rate, is_default) is **wired but functionally inert**. It's referenced (`Tax.tax_class_id` FK, class CRUD routes, the `tax_classes` exchanger, a fe-admin "Tax Classes" CRUD section, a tax-list filter) but **nothing reads the class to make a decision**: pricing (`Tax.calculate`/`is_applicable`) uses rate + country; entity assignment (S72.3) links plans/products/resources M2M to specific **`Tax`** rows, not classes; netto/brutto (S72.4) operates on the applied `Tax` rows. It's a vestige of the standard "product → tax *class* → resolve by class+country" model that S72.3 superseded with direct entity→tax links.

## Locked decision
**D1 — Flatten, don't keep, don't wire class-resolution.** With "entity → specific taxes" confirmed, collapse `TaxClass` into a denormalized **`tax_class` VARCHAR** label on `Tax` (mirrors the existing legacy `shop_product.tax_class` string). Preserves the realized use (a categorisation label + filter) while removing a model, table, FK, CRUD surface, and an exchanger. *(Not chosen: dropping categorisation entirely, or wiring class+country resolution — neither matches the current design or the "narrowest change" rule.)*

## Scope — as built
**Backend (core):**
- **Migration `20260613_1100_flatten_tax_cls`** (down_revision `20260613_1000_curr_flags`): add `tax_class VARCHAR(50)` to `vbwd_tax`; data-migrate each tax's old class → its `code`; drop `tax_class_id`; drop the `vbwd_tax_class` table. `down()` best-effort (recreates the table + FK column; class rows/links not restorable — documented). Validated up→down→up on real PG.
- **Model:** delete `TaxClass`; `Tax` drops the FK/relationship, gains `tax_class` (string, nullable); `to_dict()` updated. `calculate`/`is_applicable` untouched.
- **Routes** (`vbwd/routes/admin/tax.py`): remove the four TaxClass routes (now 404); tax create/update accept/return `tax_class` (string); list filter `?tax_class=<code>`.
- **Exchanger** (`core_exchangers.py`): remove the `tax_classes` exchanger + the `TaxesExchanger` FK-by-code subclass; `taxes` is now a plain `BaseModelExchanger` with `public_fields = [name, code, description, rate, country_code, region_code, is_active, is_inclusive, tax_class]`, natural key `code`.

**fe-admin:** remove the "Tax Classes" CRUD section + its import/export control; replace the class picker on the rate form with a `tax_class` text field (standard/reduced/zero datalist, optional); keep one `taxes` import/export.

**Bonus fix — in-use tax-delete 409 (closes an S72.3 DoD gap):** the core tax-delete route now catches the `IntegrityError` from a `*_tax` join table's `ON DELETE RESTRICT` FK → rolls back → returns **409 "Tax rate is in use and cannot be deleted"** instead of a 500. This fixes bulk tax delete (the fe-admin bulk-delete already handled 409 per-row). Regression test `test_delete_in_use_rate_returns_409_not_500` added.
