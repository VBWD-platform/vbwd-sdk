## Problem

There is no way to migrate an existing WordPress blog into the CMS. The admin needs a **"WP Import"** tab on `/admin/import-export` where they paste a WordPress RSS URL, see how many feed posts are **never imported** vs **already imported**, and import a chosen batch (10/20/50/100) of articles **as-is**: same publish dates/times, full content with **images downloaded and re-hosted locally**, featured image, tags and categories — **never creating duplicate terms** and **never importing the same post twice** (unless its imported copy was removed completely). Below the form, a sortable/searchable/paged table of imported posts with bulk operations; clicking a row opens the normal post editor.

## Locked decisions

| # | Decision |
|---|---|
| **D1 — RSS is the article source; the WP REST API only enriches.** | Articles come from the RSS feed. The pasted URL is **tokenized with its own query params** (real shape: `https://<host>/feed/?token=<secret>&show_on_page=10&show_unread_only=false&mark_as_read=false`), so the `FeedReader` treats it as an opaque base and **merges** params instead of appending naively: existing params (incl. `token`) are preserved, `show_on_page` is set to the fetch size, and `paged=N` is added to walk the feed — until an empty page, an HTTP error, the `max_feed_pages` cap, **or a page yielding no new GUIDs** (the stop condition for feeds that ignore `paged` and repeat the first page). Full body = `content:encoded`, imported **as-is** into `content_html` (no rewriting beyond image URLs, D4); excerpt = `<description>`; publish date = `pubDate` → `published_at`, status `published`. WP RSS does **not** distinguish tags from categories (both are plain `<category>`) and rarely carries the featured image — so the importer makes **one optional, cached, paginated** fetch of the site's public REST **categories** (`/wp-json/wp/v2/categories?per_page=100&page=N`, capped) — the small curated set — and classifies: name in the category set → category, **any other name → tag** (the tag set is unbounded on real WP sites — thousands of tags — so it is never fetched; proven on the live feed where the non-inverted rule misclassified everything as category). Graceful fallback when REST is unreachable (likely on token-gated hosts): all `<category>` values import as CMS categories (RSS's own semantics). Parsing via **`feedparser`** (new backend requirement) — no hand-rolled XML. |
| **D1a — Remote read-tracking is never used; the token is a secret.** | The importer **forces `show_unread_only=false&mark_as_read=false`** whatever the admin pasted: dedup authority is exclusively the local GUID ledger (D2) — relying on the feed's unread cursor would mutate remote state, break re-import-after-removal, and collide with any other consumer of the same token. The `token` query value (and any `*token*`/`*key*`-named param) is **masked everywhere it could surface**: log lines, raised errors, run summaries, and the persisted `feed_url` column (stored token-masked; the full URL exists only in-request). |
| **D2 — The WP GUID is the identity; "removed completely" reopens it.** | Model **`WpImportedPost`** (`wp_imported_post`): `wp_guid` (unique), `wp_link`, `feed_url` (token-masked, D1a), `cms_post_id` (plain UUID — **no cross-plugin FK** into `cms_post`, avoiding the known migration-graph anchoring trap [[project_migration_graph_fragmentation]]; plugin migration is its own Alembic branch in `plugins/wp_import/migrations/versions/`), `title_at_import`, `created_at` (= import datetime). A feed item is importable iff **no `WpImportedPost` row with its GUID references a still-existing CMS post**. Trash/draft status still counts as imported; only complete deletion of the post (via the tab's bulk remove, or externally via the CMS UI — staleness detected by checking post existence and purging the orphan row lazily) makes the GUID importable again. |
| **D3 — Dates: `published_at` carries the WP date; `created_at` cannot.** | `PostService.create_post` accepts `published_at` but `created_at` is BaseModel-generated and not settable — verified. So "same dates, times" = the WP `pubDate` lands in `published_at` (the publicly visible/ordering date); the import moment is tracked on `WpImportedPost.created_at` (the table's "import datetime" column). The core/cms model is **not** modified to backdate `created_at` (no overengineering; the public contract is `published_at`). |
| **D4 — Images are downloaded and re-hosted; URLs rewritten.** | Every `<img src>` in `content:encoded` is downloaded (HTTP timeout + max size from config) and uploaded through `CmsImageService.upload_image(bytes, filename, mime_type)`; the `src` is rewritten to the returned local `url_path`. The same remote URL within one run is downloaded once (in-run cache — DRY). Featured image precedence: `media:content`/`media:thumbnail` → `enclosure` → **REST `featured_media` lookup** (slug from the WP link → `posts?slug=…&_fields=featured_media` → `media/<id>?_fields=source_url`; ≤2 small public wp-json GETs per item, only when the feed signals are absent, any failure silently falls through — proven necessary on the live feed, where posts are text-only in RSS but all carry REST featured images) → **first content image**; stored re-hosted in `featured_image_url`. A failed image download does not fail the post: the original remote URL is kept and the miss is reported in the run summary. The RSS `<description>` may carry HTML — it is tag-stripped/entity-decoded to a plain-text `excerpt` (only the excerpt; `content_html` stays as-is). |
| **D5 — Terms are deduplicated via the existing taxonomy, never re-invented.** | Categories and tags resolve through `TermService.find_or_create(term_type, name)` (slug-based dedup already built) and attach via the existing post-term assignment — the importer creates **zero** new taxonomy machinery and can never double a term. |
| **D6 — No async-job framework; the frontend drives chunks.** | One `POST run` imports the next ≤ `chunk` (10) never-imported items and returns a per-item summary. The tab requests chunks repeatedly until the selected total (10/20/50/100) is reached or the feed is exhausted — progress UI for free, no job table/worker/polling infra. |
| **D7 — Bulk ops reuse the CMS admin API; only "remove" is owned here.** | Move-to-draft → existing `POST /api/v1/admin/cms/posts/bulk/status`; assign-category / add-tag → existing `POST .../bulk/assign-term` (DRY — the cms plugin already ships these). Only **remove selected imported** is a wp_import endpoint: it deletes the CMS posts completely (via `PostService`) **and** their `WpImportedPost` rows in one transaction → those GUIDs become importable again (D2). |
| **D8 — Permission: reuse `cms.manage`.** | The tab and all endpoints manipulate CMS content, so they're gated on the existing `cms.manage` (tab `requiredPermission` + `@require_admin`/`@require_permission` on routes). No new permission. |

## Scope

### Slice 1 — backend: `wp_import` plugin
- Plugin class (deps per header), `config.json` + `admin-config.json` (≥ `debug_mode`; plus `http_timeout_seconds`, `max_feed_pages`, `max_image_bytes`), registration in `plugins/plugins.json` + `plugins/config.json`; migration (own branch) for `wp_imported_post`.
- `FeedReader` (feedparser wrapper: query-param merge + forced read-tracking flags + token masking per D1/D1a, paged fetch with the no-new-GUIDs stop condition, item normalisation — guid/link/title/content/excerpt/pubDate/terms/media) and `WpTermClassifier` (cached REST lookup + fallback, D1) as separate single-purpose collaborators (SRP), injected into **`WpImportService`** (DI via container providers in `on_enable` [[project_plugin_di_provider_registration]]):
  - `stats(feed_url)` → walk feed → `{total_in_feed, never_imported, already_imported}` using the D2 importability rule (purges stale orphan rows it encounters).
  - `run(feed_url, chunk)` → import next ≤ chunk never-imported items: create post via `PostService.create_post` (type `post`, slug from the WP link path, `-N` suffix on collision), D4 images, D5 terms, write `WpImportedPost`; per-item result list (imported / skipped-duplicate / image misses / errors — an item failure skips that item, never aborts the run).
  - `list_imported(search, sort, page, per_page)` → import rows joined with live post data (title, status, category/tag names); stale rows purged, not listed.
  - `remove(import_ids)` (D7).
- Routes (`/api/v1/admin/wp-import/…`, all D8-gated): `GET stats`, `POST run`, `GET posts`, `POST posts/bulk/remove`.
- `populate_db.py`: none needed (importer is the data source); demo fixture feeds live in tests only.

### Slice 2 — fe-admin: `wp-import` plugin
- `plugins/wp-import/index.ts` (IPlugin, named export): `extensionRegistry.register('wp-import', { dataExchangeTabs: [{ id:'wp-import', label:'WP Import', component: WpImportTab, requiredPermission:'cms.manage' }] })` + translations (all locales). No new route — the tab lives inside `/admin/import-export`.
- **`WpImportTab.vue`** (+ Pinia store via `sdk.createStore`):
  - Import form: RSS URL input, batch-size select (10/20/50/100), **Check feed** → stats line ("N never imported · M already imported"), **Import** → chunked runs (D6) with progress + final summary (incl. image misses).
  - Imported-posts table: fe-core `Table.vue` + `Pagination.vue`; columns **checkbox · title · cat · tags · import datetime**; sortable columns, search box, per-page select (20/50/100); bulk action bar — **Remove selected** (confirm dialog → wp-import remove endpoint), **Move to draft**, **Assign to category** (category picker), **Add tag** (term picker/free text → find-or-create via the cms assign-term flow) (D7).
  - Row click → `router.push({ name: 'cms-post-edit', params: { id } })` — the normal post editor (entity-navigation principle, no modals).

### Deferred (explicitly out of scope)
WXR/XML-export import · author mapping (`dc:creator` → `author_id` stays null) · comments · scheduled/incremental sync (re-runs are manual) · media library dedup across runs (in-run cache only) · importing WP pages (posts only).
