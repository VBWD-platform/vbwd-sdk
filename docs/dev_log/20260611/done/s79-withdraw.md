# S79 — Withdraw (token balance → money payout via payment plugins)

**Area:** NEW plugin duo `withdraw` (`vbwd-backend/plugins/withdraw` + `vbwd-fe-user/plugins/withdraw`, standalone repos `vbwd-plugin-withdraw` / `vbwd-fe-user-plugin-withdraw`) + payout capability added to backend payment plugins `stripe`, `paypal`, `truemoney` (the "TruePay" of the ask), `promptpay` · **Depends on:** core token ledger (`vbwd/models/user_token_balance.py`, `vbwd/services/token_service.py`), `PaymentProviderPlugin` contract (`vbwd/plugins/payment_provider.py`), fe-user `userNavRegistry`.
**Engineering requirements:** TDD-first · DevOps-first · SOLID · DI · DRY · Liskov · clean code · **NO OVERENGINEERING** · drop deprecated code in touched paths — **BINDING per [`_engineering_requirements.md`](../sprints/_engineering_requirements.md)** (which inherits the 20260525 canonical statement). Quality guard: `bin/pre-commit-check.sh --full` green on backend (`--plugin <name> --quick` while iterating, full run incl. every touched payment plugin) + fe-user Vitest/ESLint green. **Not committed** without explicit instruction.

## Problem

A user can accumulate tokens (purchases, bonuses, refunds — core `vbwd_token_transaction` ledger) but there is **no way to get money out**. Nothing named withdraw/payout exists anywhere in core or plugins today (only meinchat's peer-to-peer token transfer, which is unrelated). The user needs a **"Withdraw"** item in the dashboard burger menu opening a page that shows the **token balance** and the list of **payment methods that support withdraw**, where the user enters a destination account (PayPal email, Stripe connected account, TrueMoney wallet, PromptPay ID) and an amount, and the platform sends the money.

## Locked decisions

| # | Decision |
|---|---|
| **D1 — Payout is an opt-in capability ABC in core plugin infra.** | `vbwd/plugins/payment_provider.py` (which already owns `PaymentProviderPlugin`) gains a sibling ABC **`PayoutProvider`**: `get_payout_destination_schema() -> list[dict]`, `create_payout(amount, currency, destination, reference_id) -> PayoutResult`, `get_payout_status(provider_payout_id) -> str`. Core only **defines the contract** (exactly like `PaymentProviderPlugin`) and never calls it — the withdraw plugin discovers implementers via `PluginManager.get_enabled_plugins()` + `isinstance`. Narrow port (ISP): only these three methods. Liskov: every implementer (and the test fakes) honours the same semantics — `create_payout` either returns a `PayoutResult` or raises a typed payout error; no provider-specific surprises leak to the caller. |
| **D2 — Orchestration lives in a NEW `withdraw` plugin, not core.** | Core stays agnostic: no withdraw routes/services in `vbwd/`. The withdraw plugin owns the `WithdrawRequest` model, the service, the API and the migration (in `plugins/withdraw/migrations/versions/`, registered in `alembic.ini` `version_locations`). New-convention layout: source in `plugins/withdraw/withdraw/` (plugin id, not `src/`). Ships `config.json` + `admin-config.json` with at least `debug_mode`. |
| **D3 — Balance source is a 3-method seam, token-backed in v1.** | The withdraw plugin defines **`IWithdrawableBalance`**: `get_balance(user_id) -> (amount, currency)`, `debit(user_id, amount, reference_id)`, `refund(user_id, amount, reference_id)`; sources register in a plain dict registry keyed by `balance_source` (v1 registers only `"tokens"`). `WithdrawRequest` carries a `balance_source` column (default `"tokens"`). This is the one deliberate seam beyond today's need: S80 (GHRM marketplace "My money") plugs its earnings ledger into exactly this registry — one column + one dict now spares S80 a migration and a refactor. Nothing else is speculative. |
| **D4 — Token→money conversion is config.** | `TokenWithdrawableBalance` converts at `token_to_currency_rate` × tokens in `currency`, with `min_withdraw_tokens` floor — all three in the withdraw plugin `config.json`. No fees in v1. Debit/refund go through core `TokenService.debit_tokens`/`credit_tokens` (DI via container — never raw SQL), `reference_id` = the withdraw request id. |
| **D5 — `WITHDRAW` joins the core token-ledger vocabulary.** | `TokenTransactionType` gains `WITHDRAW` (generic ledger vocab alongside PURCHASE/REFUND/ADJUSTMENT — tokens are core-owned, so this is not a plugin-vocab leak; agnosticism oracles must stay green). Verify the column type first: if it's a native PG enum a **core** migration adds the value (must resolve standalone, plugin-free); if it's a varchar, no migration. |
| **D6 — Money out requires admin approval; failure refunds.** | Status flow `pending → approved → processing → completed`, terminal failures `failed` / `rejected`. Balance is debited at request time (prevents double-spend); `failed`/`rejected` triggers `refund()` (token credit with `WITHDRAW` reversal entry). `require_admin_approval` config defaults `true`; `false` short-circuits straight to payout (dev/demo only). v1 admin surface is **API-only** (no fe-admin page — deferred, see below). |
| **D7 — The destination form is schema-driven; no fe payment-plugin changes.** | `GET /withdraw/providers` returns each payout-capable provider with its `destination_schema` (field name/type/label i18n key, e.g. paypal → `[{name:"email", type:"email"}]`). The fe-user withdraw plugin renders the form generically from the schema — the four `*-payment` fe plugins are untouched (DRY, OCP). |
| **D8 — Reusable fe panel.** | The fe-user withdraw plugin registers **`WithdrawPanel`** (props: `balanceEndpoint`, `balanceSource`) via `sdk.addComponent` so S80's "My money" page embeds the identical provider-list + destination-form + history UI without importing across plugin directories. |

## Scope

### Slice 1 — backend: port + withdraw plugin
- `vbwd/plugins/payment_provider.py`: `PayoutProvider` ABC + `PayoutResult` dataclass + typed `PayoutError` (D1). Core agnosticism oracle stays green.
- `TokenTransactionType.WITHDRAW` (+ core migration only if PG enum, D5).
- `plugins/withdraw/`: plugin class (`get_blueprint`, `get_url_prefix() -> ""` — public + admin prefixes), `IWithdrawableBalance` + registry + `TokenWithdrawableBalance` (D3/D4), model **`WithdrawRequest`** (`withdraw_request`): `user_id` FK→`vbwd_user.id` `ON DELETE CASCADE`, `balance_source`, `amount` (source units, e.g. tokens), `payout_amount`, `currency`, `provider`, `destination` (JSONB), `status`, `provider_payout_id`, `error`, timestamps — `BaseModel` pattern, explicit `to_dict()`. Repo + `WithdrawService` (validate min + balance → debit → request; approve → `create_payout` → status; reject/fail → refund) wired as DI providers in `on_enable` ([[project_plugin_di_provider_registration]] — validate the full runtime path, container providers included).
- Routes: `GET /api/v1/withdraw/providers` · `POST /api/v1/withdraw` · `GET /api/v1/withdraw/requests` (auth, self-service) · admin `GET /api/v1/admin/withdraw/requests`, `POST /api/v1/admin/withdraw/requests/<id>/approve|reject`.
- Register in `plugins/plugins.json` + `plugins/config.json`; demo rows via `populate_db.py` through the service layer (never raw SQL).

### Slice 2 — backend: provider implementations (each its own red→green loop)
- **paypal:** `PayPalPlugin(PaymentProviderPlugin, PayoutProvider)`; `PayPalSDKAdapter.create_payout_batch` (Payouts API v1, destination = email).
- **stripe:** `StripePlugin` + `StripeSDKAdapter.create_transfer` (Connect Transfers, destination = `acct_…` connected-account id).
- **truemoney:** `TrueMoneyPlugin` + `TrueMoneySDKAdapter` wallet transfer (HMAC-signed, destination = msisdn).
- **promptpay:** `PromptPayPlugin` implements `PayoutProvider` via a new `create_funds_transfer` seam on `bank_clients/base.py` (KBank/SCB impls, destination = PromptPay ID); PromptPay is inbound-QR today, so outbound rides the existing bank-client abstraction.
- All adapters mocked in unit tests (MagicMock, per house test pattern); no live calls in CI. Drop any deprecated/dead adapter code encountered while touching these files instead of extending it.

### Slice 3 — fe-user: withdraw plugin
- `vbwd-fe-user/plugins/withdraw/index.ts` (named export, per convention): `userNavRegistry.register({ pluginName:'withdraw', to:'/withdraw', labelKey:'withdraw.title', testId:'nav-withdraw' })` — top-level sidebar item (the sidebar is what the mobile burger opens); route `/withdraw` (auth-guarded); Pinia store via `sdk.createStore`; i18n for all locales.
- `WithdrawView.vue` = thin wrapper around **`WithdrawPanel`** (D8): token balance (`GET /api/v1/user/tokens/balance`) with computed money preview (rate from the providers payload), provider radio list, schema-driven destination form (D7), amount + submit, own request history with statuses.
- Entity navigation principle: a request row links to a canonical detail view (`/withdraw/:id`), no modals.

### Deferred (explicitly out of scope)
fe-admin approval UI (admin uses the API this sprint) · fees/limits beyond `min_withdraw_tokens` · payout webhooks/async status polling (v1 treats provider acceptance as `processing→completed`; status refresh via `get_payout_status` on read) · KYC.

## Definition of done
A user opens **Withdraw** from the dashboard burger menu, sees the token balance, picks one of the payout-capable methods (PayPal, Stripe, TrueMoney, PromptPay), enters the destination + amount, and the request is debited, admin-approved, and paid out via the provider (mock in CI), with refund on failure/rejection. Core gains only the `PayoutProvider` contract + `WITHDRAW` ledger vocab; all orchestration sits in the `withdraw` plugin; the four fe `*-payment` plugins are untouched. `bin/pre-commit-check.sh --full` green on backend (incl. each touched payment plugin scope) + fe-user Vitest/ESLint green; agnosticism oracles green; no commits unless explicitly instructed.
